# 動作仕様

## 重要: 「既存 worktree なら移動するだけ」の挙動は `enter` に移行しました

過去のバージョンでは、`create <name>`（＝ `wt <name>`）は worktree ルートが既に存在する場合に
リポジトリ選択・作成処理を丸ごとスキップし、`after` フックだけを実行していました（cmux 等への
自動遷移フックと組み合わせて「既にあるなら、ただそこへ移動するだけ」にするためのショート
サーキットです）。

**このショートサーキットは廃止されました。** 現在の `create` は常に冪等な差分作成を行い
（後述）、リポジトリ選択・fetch・worktree 作成を毎回試みます。「ただ遷移したいだけ」の
ユースケースは、明示コマンド [`wt enter <name>`](Usage#enter--既存-worktree-への遷移) に
切り出されています。`enter` は worktree ルートの存在だけを確認し、`after` フックだけを
実行します（新しいリポジトリの追加や作成は一切行いません）。

cmux などへの遷移フックを設定している場合、そのフックの設定自体（`[[hooks.after]]`）は
変更不要です。運用を次のように変えるだけで、以前と同じ体感が得られます:

- 「その worktree に移動したいだけ」のとき → `wt enter <name>`
- 「worktree を作成（または未作成分を追加）してから移動したい」とき → `wt create <name>`
  （＝ `wt <name>`。`after` フックは作成完了後にも実行されます）

専用の `on_enter` フックタイミングは追加していません。`after` は「作成完了 / 遷移」の
両方を意味するフックであり、`create` の完了時にも `enter` の実行時にも呼ばれます。

## `create` の冪等性・差分作成

`create <name>` は**冪等**です。ショートサーキット（全スキップ）は行わず、次のロジックで
「未作成分だけを差分で追加」します:

- `before` フックは、対象が既に全て作成済みかどうかによらず**常に実行**されます。
- `--repo` / `--all` で対象を明示した場合、この worktree に既に存在するリポジトリは
  "スキップ（worktree は既に存在します）" として報告されるだけで、失敗にはなりません。
  `--repo` に指定した名前が `repos_dir` に見つからない場合はエラーです。
- 対話選択（`--repo` / `--all` のどちらも省略、かつ標準入出力が端末）の場合は、
  **この worktree にまだ存在しないリポジトリだけ**が選択肢に並びます（差分選択）。
  探索されたリポジトリが 1 件以上あるのに、選択肢が 1 件も無い（＝全リポジトリが
  作成済み）場合は「追加するリポジトリはありません」と表示して正常終了します
  （`after` フックは実行されます）。
- 非対話（パイプ・CI・MCP など）で `--repo` / `--all` のどちらも指定しない場合は、
  「--repo か --all を指定してください」というエラーになります（対話プロンプトへは
  進みません）。

新しく worktree を実際に作成したリポジトリについては、そのリポジトリ・そのサーバーの
setup 記録（[Server Management](server-management.md) を参照）が無効化されます。これにより
`remove` で削除して同じ名前で作り直した場合、setup（依存インストールなど）が正しく
再実行されます。

## リポジトリ検出

`repos_dir` 直下のみ（再帰なし）を検査します。`.git`（ディレクトリ／ファイルのどちらでも
可）を持つものが対象です。名前順にソートして表示・処理します。

## worktree 名の規則

worktree 名はブランチ名としてもディレクトリパスとしても使われるため、`git
check-ref-format` に準拠した規則で検証されます:

- `/` で区切られたセグメントの並びとして扱われます（`feature/login` のような階層的な
  名前が使えます）。
- 各セグメントに使えるのは英数字・`.`・`_`・`-` のみです。
- 拒否される例: 空のセグメント（先頭・末尾・連続した `/`）、`.` / `..` のみのセグメント、
  セグメント先頭の `-`（git がオプションと解釈するため）や `.`、`.lock` で終わるセグメント
  （git の参照ロックファイルと衝突するため）、空白・制御文字・`: ? * [ \ ~ ^` および `@{`
  を含むもの。
- 許可される例: `feat_login`、`v1.2`、`ABC-123`、`feature/login`。

エラーメッセージには具体的な違反理由（どの文字・どのセグメントが不正か）が含まれます。
`--repo` に渡すリポジトリ名は worktree 名とは別の規則（`/` を含まない単一セグメント）で
検証されます。

## 並列処理

選択されたリポジトリは、並列度で上限を設けた goroutine（セマフォ）で処理されます。1
リポジトリが失敗しても他のリポジトリの処理は止まりません。進捗出力は直列化されており
行が混ざりません。結果はリクエスト順に集約されます。`Ctrl-C`（SIGINT）や SIGTERM は
`context.Context` のキャンセルとして伝播し、新規着手中のリポジトリは実行中の git
プロセスが終了させられ、未着手のリポジトリは「キャンセル」としてスキップに集約されます。

**並列実行数の自動決定**: `-j`（`--concurrency`）未指定時（または `0`）は、選択した
リポジトリ数と CPU コア数から自動で決まります。処理は fetch（ネットワーク）とディスク
I/O が主体で CPU バウンドではないため、コア数より多めに並列化します。具体的には
`min(選択リポジトリ数, (CPU コア数 × 4) を 4〜16 にクランプした値)` を上限とします。
`-j` を指定した場合はその値が上限となり（同様にリポジトリ数でクランプ）、自動決定より
優先されます。優先順位はフラグ `-j` ＞ 環境変数 `WT_CONCURRENCY` ＞ 設定ファイルの
`concurrency` ＞ 自動決定です。

## ベースブランチの解決

各リポジトリのベースブランチは次の優先順位で解決されます:

```
--base フラグ（MCP: base パラメータ） ＞ [repos.<repo>].base ＞ [defaults].base ＞ "auto"
```

`auto`（既定値）は `git symbolic-ref refs/remotes/<remote>/HEAD` → `main` の存在確認 →
`master` の存在確認、の順にローカルの ref だけを見て解決します（ネットワークアクセスは
発生しません）。

fetch は `git fetch --no-tags <remote> <base>` で対象ブランチ 1 本だけに限定し、
無関係な他ブランチの更新を避けて高速化します。**fetch に失敗しても即座に失敗とはせず**、
`refs/remotes/<remote>/<base>` という既存の追跡 ref が既にあれば、それを使って作成を
続行します（オフライン・ネットワーク不調時の degrade。進捗に「fetch に失敗したため、
既存の追跡ブランチから作成します」と表示されます）。既存の追跡 ref も無い場合のみ
失敗になります。

## 既存の作成先ディレクトリの扱い

作成先（`<worktrees_dir>/<name>/<repo>/`）に既に何かがある場合、一律スキップではなく
中身を見て判定します:

- **空ディレクトリ**: 残骸とみなして削除し、そのまま作成します。
- **既存の worktree（`.git` がある）**: 「既存」としてスキップします（警告のみ、失敗扱い
  にしない）。これが冪等な差分作成の基盤です。
- **無関係な中身がある / ディレクトリではない**: 上書きせず**失敗**として扱います。

## 同名ブランチが既に存在する場合

衝突として**失敗**させます（履歴が異なる可能性があるため、黙って再利用しません）。
削除するか別名を使ってください。ネットワーク fetch の前に検査するため、オフラインでも
即座に失敗します。

## 手動削除した worktree の後始末

`git worktree remove` ではなく `rm -rf` でディレクトリだけ消した場合に残る
`.git/worktrees/<id>` メタデータは、同名で作り直す際に自動で prune（除去）します
（作業ツリーが残っている稼働中の worktree には触れません）。`wt doctor` はこの手の
残骸（設定/状態/ログの不整合を含む）を一括で発見・修復します。

## 認証

fetch はローカルの `git` コマンドに委譲するため、ユーザーの git 設定がそのまま使われます
（`SSH_AUTH_SOCK` の SSH agent、credential helper など）。対話的な認証プロンプトは無効化
（`GIT_TERMINAL_PROMPT=0`）しているので、認証情報が無い場合はハングせず即座に失敗します。
`file://` や公開 https remote は認証なしで足ります。

## キャンセルと終了コード

`Ctrl-C`（SIGINT）・`SIGTERM` は `context.Context` のキャンセルへ変換され、実行中の
git・フック・サーバーのライフサイクルコマンドまで伝播します（デタッチ起動されたサーバー
本体は仕様として対象外です）。対話選択プロンプトの中断（Ctrl-C）は「空選択（何も選ばず
Enter）＝何もしない正常終了」とは区別され、いずれもシェルの `128+SIGINT` 慣習に合わせて
**exit 130** になります。通常のエラーは exit 1、成功は exit 0 です。
